# 接口设计规范

## 一、路径设计

- 统一前缀：`/api/v1`
- 资源命名使用小写加中划线或复数名词
- 避免把动词堆到路径里，优先用 HTTP Method 表达语义

示例：

- `GET /api/v1/users`
- `GET /api/v1/users/{id}`
- `POST /api/v1/orders`
- `PUT /api/v1/orders/{id}`

## 二、请求约定

### 查询接口

- 分页统一使用 `pageNum`、`pageSize`
- 排序统一使用 `sortField`、`sortOrder`
- 模糊搜索统一使用 `keyword`

### 写接口

- `POST` 用于新增
- `PUT` 用于全量更新
- `PATCH` 用于局部更新
- `DELETE` 用于删除或软删除入口

## 三、响应约定

统一结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

建议：

- 列表接口统一返回分页对象
- 枚举值同时提供 code 和 label
- 布尔状态优先返回明确字段，如 `enabled`、`deleted`

## 四、兼容性规则

- 已上线字段不要随意删除
- 字段重命名优先走兼容期，而不是直接替换
- 版本升级时，用新版本路径或新字段承载变更

## 五、错误处理

- 参数错误：`400`
- 未登录：`401`
- 无权限：`403`
- 资源不存在：`404`
- 服务异常：`500`

错误信息要求：

- 能让前端明确展示
- 能让后端定位问题
- 不泄露内部实现细节

## 六、变更清单

接口改动后至少同步检查：

1. Swagger / Knife4j 是否更新
2. 前端请求方法和类型定义是否更新
3. DTO / VO / Entity 是否一致
4. 数据库字段和索引是否需要调整
5. 文档、示例请求和错误码说明是否更新
