# 团队协作规范（前后端共用，落地版）
## 一、Git分支管理规范
### 1. 分支类型及用途
| 分支类型 | 分支名称格式 | 用途 |
|----------|--------------|------|
| 主分支   | main         | 生产环境代码，仅接收release分支合并，禁止直接提交 |
| 开发分支 | develop      | 开发环境主分支，所有开发完成后合并到此分支，用于联调 |
| 功能分支 | feature/xxx  | 单个功能开发（xxx为功能描述，小驼峰），从develop分支创建，完成后合并回develop |
| 修复分支 | bugfix/xxx   | 开发环境bug修复（xxx为bug描述），从develop分支创建，完成后合并回develop |
| 发布分支 | release/xxx  | 版本发布准备（xxx为版本号，如1.0.0），从develop分支创建，测试通过后合并到main和develop |
| 紧急修复 | hotfix/xxx   | 生产环境紧急bug修复，从main分支创建，完成后合并到main和develop |

### 2. 分支操作规范
1.  所有分支创建前，需确认无重复分支，命名符合格式要求
2.  功能/修复分支开发完成后，提交MR/PR，由团队负责人审核通过后再合并
3.  合并前需拉取最新分支代码，解决冲突，确保代码可运行、符合编码规范
4.  分支合并后，及时删除对应功能/修复分支，避免分支冗余

## 二、Git提交规范
### 1. 提交信息格式
```
type(scope): description
```
- type：提交类型（必填）
  - feat：新增功能
  - fix：修复bug
  - docs：修改文档（不涉及代码）
  - style：代码格式调整（不影响代码逻辑）
  - refactor：代码重构（不新增功能、不修复bug）
  - test：新增/修改测试用例
  - chore：构建、依赖、脚本等辅助操作
- scope：提交范围（可选，如frontend、backend、api、db）
- description：提交描述（简洁明了，100字以内，说明做了什么）

### 2. 提交示例
```
feat(frontend): 新增用户登录页面组件
fix(backend): 修复用户列表查询分页异常
docs(global): 更新通用接口文档状态码说明
```

## 三、前后端联调规范
1.  后端接口开发完成后，先自测通过（使用Postman/Swagger），再通知前端联调
2.  联调过程中，接口变更需及时同步对方，避免无效开发
3.  联调遇到问题，先各自排查（前端看请求参数、响应数据；后端看日志），无法解决再协同排查
4.  联调完成后，共同测试核心场景，确认无异常后，再提交代码到对应分支

## 四、文档维护规范
1.  全局文档（.claude-global/docs）由团队负责人统一维护，修改后通知所有成员
2.  项目独有文档（前端/后端docs目录）由对应开发人员维护，确保内容与代码一致
3.  文档修改需遵循“谁修改、谁更新”原则，避免文档与实际代码脱节
4.  文档命名、格式统一，便于查找和维护

## 五、日常协作规范
1.  每日同步开发进度，遇到问题及时沟通，避免延误项目进度
2.  开发过程中，遵循编码规范、命名规范，确保代码可维护性
3.  复用已有代码/组件，不重复开发，提升开发效率
4.  协助团队成员解决问题，共同提升团队技术水平
