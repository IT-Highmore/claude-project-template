# 前端组件开发规范（Vue3+TS）
## 一、组件分类及命名规范
### 1. 组件分类
- 基础组件（common组件）：通用、可复用性强，无业务逻辑（如Button、Input、Card），放在src/components/common
- 业务组件：与具体业务相关，可复用（如UserForm、OrderList），放在src/components/对应业务模块（如user、order）
- 页面组件：页面级组件，不可复用，放在src/views目录，每个页面一个组件，可包含多个业务组件

### 2. 命名规范
- 组件文件：大驼峰命名（如UserForm.vue、Button.vue），禁止小写、下划线命名
- 组件名称：与文件名一致，大驼峰，禁止使用中文、特殊字符
- 基础组件：前缀统一为Base（如BaseButton.vue、BaseInput.vue），便于区分
- 业务组件：按业务模块+功能命名（如UserForm.vue、OrderTable.vue）
- 页面组件：按页面功能命名（如UserList.vue、Login.vue）

## 二、组件编写规范
### 1. 基础结构
每个组件需包含模板（<template>）、脚本（<script setup lang="ts">）、样式（<style scoped&gt;），结构清晰：
```vue
&lt;template&gt;
  <!-- 模板内容，结构清晰，缩进4个空格 -->
  &lt;div class="user-form"&gt;
    <!-- 组件内容 -->
  </div>
</template>

<script setup lang="ts">
// 脚本内容，导入依赖、定义类型、编写逻辑
import { ref, computed } from 'vue'
// 导入其他组件
import BaseInput from '@/components/common/BaseInput.vue'

// 定义props类型
interface Props {
  userId?: number
  title: string
}

// 定义props
const props = defineProps<Props>()

// 定义emits
const emit = defineEmits(['submit', 'cancel'])

// 组件逻辑
const formData = ref({})
const handleSubmit = () => {
  emit('submit', formData.value)
}
</script>

<style scoped lang="scss">
// 样式内容，使用scoped避免污染，优先使用Tailwind
.user-form {
  @apply p-4 bg-white rounded-lg;
  // 自定义样式
}
</style>
```

### 2. Props规范
- 所有props必须定义类型，使用interface或type定义，禁止不定义类型
- props命名：小驼峰，与后端字段、父组件传递参数保持一致
- 必传props：添加required: true，可选props：设置默认值
- 避免props传递过多数据，复杂数据可使用Pinia全局状态，或拆分组件
- 禁止通过props修改父组件数据（单向数据流），需通过emit触发父组件修改

### 3. Emits规范
- 命名：小驼峰，动词+名词（如submitForm、cancelEdit），清晰说明触发事件的用途
- 定义：使用defineEmits明确声明，禁止隐式触发事件
- 传参：emit传参需明确类型，避免传递过多参数，复杂参数可封装为对象
- 避免过度使用emit，可通过Pinia实现组件间通信（跨组件、非父子组件）

### 4. 模板规范
- 缩进：统一4个空格，结构清晰，嵌套层级不超过4层（避免过度嵌套）
- 标签使用：优先使用Vue3内置组件，避免使用原生标签（如div、span）过度嵌套
- 指令使用：v-if/v-else-if/v-else 需逻辑清晰，v-for需绑定key（唯一值）
- 样式绑定：优先使用Tailwind工具类，自定义样式需遵循BEM命名规范
- 禁止事项：禁止在模板中编写复杂逻辑（如表达式运算），复杂逻辑放在script中处理

### 5. 样式规范
- 组件样式：必须添加scoped属性，避免样式污染
- 样式优先级：Tailwind工具类 > 自定义SCSS样式，避免使用!important
- 全局样式：禁止在组件内编写全局样式，全局样式放在src/assets/styles目录
- 响应式样式：使用Tailwind的响应式类（如sm:、md:、lg:），适配不同屏幕尺寸
- 样式冗余：禁止重复编写样式，可提取SCSS混合样式、公共样式

## 三、组件复用规范
1.  基础组件：通用功能（如按钮、输入框）提取为基础组件，统一样式、功能，避免重复开发
2.  业务组件：多个页面共用的业务逻辑（如用户表单、订单表格）提取为业务组件，参数化适配不同场景
3.  组件复用原则：
    - 单一职责：一个组件只做一件事，避免组件功能过于复杂
    - 可配置性：复用组件需支持props配置，适配不同使用场景
    - 可维护性：组件逻辑清晰，注释完整，便于后续修改和扩展

## 四、组件注释规范
1.  组件注释：在script顶部添加注释，说明组件功能、用途、作者、创建时间
2.  Props注释：给每个props添加注释，说明参数含义、类型、默认值
3.  Emits注释：给每个emit添加注释，说明触发时机、传递参数含义
4.  复杂逻辑注释：组件内复杂逻辑（如表单校验、接口请求），关键步骤添加注释
5.  注释格式：简洁明了，避免冗余，使用// 注释单行，/* */ 注释多行

## 五、注意事项
1.  组件开发完成后，需自测（功能、样式、兼容性），再提交代码
2.  避免组件过大、功能过多，复杂组件可拆分为多个子组件
3.  组件命名、props命名、emit命名，需符合全局命名规范
4.  与后端联调的组件（如表单、列表），字段名需与后端保持一致
5.  组件样式需统一，与Element Plus风格保持一致，避免自定义样式冲突
